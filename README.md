## چالش‌ها و تصمیمات معماری

بزرگ‌ترین چالش این پروژه خود پیاده‌سازی نبود، بلکه تحلیل نیازمندی‌ها و انتخاب معماری مناسب بود. مستند پروژه که به صورت فایل Word ارائه شده بود، ساختار منسجم و شفافی نداشت و در برخی بخش‌ها امکان برداشت‌های متفاوت از نیازمندی‌ها وجود داشت. به همین دلیل بخش قابل توجهی از زمان صرف تحلیل مستند، بررسی سناریوهای مختلف و طراحی ساختاری شد که علاوه بر پاسخ‌گویی به نیازهای پروژه، قابلیت توسعه و نگهداری مناسبی نیز داشته باشد.

یکی دیگر از چالش‌ها، محدودیت ساختار پوشه‌های ارائه‌شده در مستند بود. اگرچه ساختار کلی پروژه مشخص شده بود، اما محل قرارگیری برخی اجزای معماری مانند Interface Markerها (نظیر `ICommand` و `IQuery`) مشخص نشده بود. در ابتدا تصمیم داشتم لایه‌ای مجزا با عنوان **Abstractions** برای نگهداری این اجزا ایجاد کنم، اما با توجه به الزام رعایت ساختار تعیین‌شده در مستند، در نهایت از ایجاد این لایه صرف‌نظر کردم و این Interfaceها را در ساختار موجود پروژه قرار دادم.

چالش دیگر انتخاب مکانیزم مناسب برای مدیریت Eventها بود. در ابتدا چند رویکرد مختلف از جمله استفاده از RabbitMQ، استفاده از MediatR و همچنین پیاده‌سازی مبتنی بر قابلیت‌های داخلی C# مورد بررسی قرار گرفت. در نهایت برای Eventهای داخلی پروژه از **MediatR** استفاده شد، زیرا تمامی مصرف‌کننده‌های این Eventها در همان سرویس قرار داشتند و استفاده از Message Broker در این سناریو پیچیدگی غیرضروری ایجاد می‌کرد. با این حال، **RabbitMQ** نیز با استفاده از **MassTransit** به طور کامل راه‌اندازی، پیکربندی و تست شد تا زیرساخت لازم برای توسعه پروژه و استفاده در سناریوهای Microservice فراهم باشد.

در نهایت، فایل `Program.cs` در لایه API برخلاف رویه‌ای که معمولاً در پروژه‌های بزرگ برای تفکیک تنظیمات به Extension Methodها انجام می‌شود، به صورت یکپارچه نگه داشته شد. دلیل این تصمیم، کوچک بودن ابعاد پروژه و خوانایی بیشتر فرآیند راه‌اندازی (Bootstrap) در قالب یک فایل بود.
